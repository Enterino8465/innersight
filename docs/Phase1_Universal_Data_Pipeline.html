Overview

Phase 0 is complete (18 commits). The current data pipeline (backend/data/pipeline.py) only loads one CERT version (r4.2). The CERT dataset has 10 versions (r1 through r6.2) with significant schema drift — missing columns, renamed activities, different date formats, absent file types, and sizes ranging from 0.5 GB to 94 GB.

Phase 1 builds an adapter layer so any version loads through one interface, producing identical canonical DataFrames. This is the foundation every downstream module depends on.

Execution method: Copy the prompt from each task and paste it into Claude Code. After each task, verify the commit was made before moving to the next.

Important: Execute tasks IN ORDER. Task 1 creates the shared schema that all subsequent tasks import.

Coding standards: Two skills are enforced. python-standards: type every signature (X | None not Optional), pathlib over os.path, logging over print, frozen dataclasses for config, explicit errors with context. ml-training: applies to dataset/tensor code in later phases.

Essential Reference: DATASET_CONTEXT.md

The uploaded DATASET_CONTEXT.md (also called CERT_DATASET_REFERENCE.md) documents every schema quirk across all 10 versions. Claude Code must read this file before implementing any adapter. It contains exact CSV column lists, date format variations, activity name differences, and the adapter fingerprint decision tree. The prompts below reference it — make sure it is available in the project root or uploaded to the conversation.

Task 1: Canonical Schema + Config Updates

The Problem

No single source of truth for what columns each log type should have after normalization. The existing pipeline.py hard-codes column expectations inline. Without a shared contract, every adapter would independently invent column names.

Also, config.py is missing CERT-specific constants: version family mappings, the real job-search domains from the dataset (the existing JOB_KEYWORDS is incomplete), and chunked-reading config.

How This Achieves the Goal

schema.py becomes the import every adapter, loader, and test uses. validate_dataframe(df, schema) catches schema drift at load time, not silently in a training loop hours later.

Task 2: Individual File Loaders

The Problem

The current pipeline calls pd.read_csv(path, parse_dates=['date']) on everything. This fails on: r1 http.csv (no header row), r2 dates (not zero-padded), r5/r6 extra columns. Each file type needs a dedicated loader that handles its parsing quirks.

Additionally, LDAP (directory of monthly CSVs), psychometric, and decoy_file each have unique formats with no existing loaders.

How This Achieves the Goal

Loaders are the lowest-level building blocks. They know HOW to parse a file but NOT which version's quirks to handle (that's the adapter's job, Task 3). An adapter configures a loader with "read this file with these column overrides" and the loader does the mechanical work.

Task 3: Version Adapters (6 Families)

The Problem

The 10 CERT versions have significant schema drift. The key differences are documented in DATASET_CONTEXT.md sections 3 and 5. Each adapter must normalize its version's raw CSVs into the canonical schemas from Task 1.

Critical Quirks (from DATASET_CONTEXT.md — Claude Code MUST read this file)

FamilyKey Differences
r1User IDs prefixed DTAA/ (strip it). HTTP has NO header row (5 cols). No email.csv, file.csv, psychometric.csv. LDAP has capitalized column names.
r2Device uses Insert/Remove not Connect/Disconnect. Email has only 4 columns (id,date,to,from). Dates not zero-padded.
r3xEmail has 7 columns — no user or pc column. User must be inferred from the from field via LDAP email→user_id mapping later.
r4xFull email schema. File has 6 columns (every row = removable media copy, no explicit flag). This is our primary training version.
r5xDevice adds file_tree. Email adds activity. File adds to_removable_media. LDAP adds projects. Decoy files exist.
r6xHTTP adds activity column. HUGE file — must use streaming. Email changes Receive→View. Content is full sentences not bag-of-words.

How This Achieves the Goal

After this task, get_adapter('r5.2').load_logon(data_dir) returns the same canonical schema as get_adapter('r1').load_logon(data_dir). Downstream code never needs to know which version it's working with.

Task 4: Answers Parser

The Problem

The current load_labels() returns a flat set of (user, date) tuples. This loses scenario info, attack start/end timestamps, and per-insider event logs. Phase 2 needs attack windows for overlap-ratio labeling. Phase 3+ needs per-scenario evaluation.

The answers directory also has two formats: flat CSVs (r2, r3, r6) and subdirectory-organized files (r4.2, r5.2). The current parser doesn't handle subdirectories.

Version Mismatch Design Decision (Council-Reviewed)

auto_detect_version() returns a FAMILY string ('r4x') because CSV fingerprinting can't distinguish r4.1 from r4.2. But insiders.csv has exact versions (dataset='4.2'). The council reviewed this and the verdict is:

When load_insiders receives a family string ending in 'x', use prefix matching against the dataset column (e.g., '4x' → match rows starting with '4'). This is safe because each data directory's insiders.csv only contains its own version's rows. Also: raise ValueError if zero insiders are found when insiders.csv exists and has rows — silent zero-labels is a critical bug.

How This Achieves the Goal

Downstream modules get structured InsiderRecord objects with attack windows instead of raw tuples. The attack window dictionary enables overlap-ratio labeling in Phase 2. Per-scenario tracking enables per-scenario evaluation in Phase 3+.

Task 6: Feature Store (Parquet Persistence)

The Problem

The pipeline recomputes features from raw CSV on every run. For r4.2 this takes ~30 seconds. For r6.2 with 85M HTTP rows, potentially hours. Feature computation is deterministic — recomputation is pure waste. Phase 2 will add EMA baselines on top, making this even more expensive.

How This Achieves the Goal

Once features are computed, they persist as Parquet. Subsequent runs load in milliseconds. Phase 2 extends this with deviation caching.
