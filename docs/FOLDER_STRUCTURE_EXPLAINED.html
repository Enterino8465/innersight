InnerSight — Folder Structure Explained

A guided map of every folder in the project, what it is for, and how it fits the four-module pipeline. Built for exam prep: read top-to-bottom once, then use it as a lookup.

Repo root: ~/Developer/innersight Python backend / ML TS/React frontend Config build & CI Data datasets Generated artifacts
The mental model in one sentence.

The repo splits into three top-level pieces: backend/ (the Python brain — data pipeline + ML + API), frontend/ (the React web UI the teacher will click through), and data/ (the bundled demo dataset it runs on). Everything else at the root — docs/, Makefile, docker-compose.yml, .github/ — is glue that builds, runs, documents, or tests those three.

③ backend/innersight/ the package — the heart of the project

This is the codebase you must know line-by-line. It is organized to mirror the four-module pipeline: ingest → features → models → scoring, plus the API that ties it together.

Three files sit at the top level of the package:
FilePurpose
api.pyThe Flask REST API — the single most important file to know. Every button in the UI calls an endpoint here (alerts, employees, investigation, suspect search, training, model comparison). Contains _generate_demo_alerts() (the live statistical scoring path) and _get_model() (loads a trained .pt if one exists, otherwise logs "No trained model found").
config.pyCentral configuration: file paths, business-hours window (07:00–19:00), internal domain (dtaa.com), risk thresholds, the list of 18 features. Change a constant here and the whole pipeline follows.
schema.pyDefines the canonical column layout every dataset version is normalized to (logon/device/email/http/file/LDAP/psychometric). The adapters target this schema.

M0 data/ — read & normalize the logs Python

backend/innersight/data/

Module 0 / Stage 1: get raw CERT CSVs into a clean, uniform table. This folder is the reason the project can handle all 10 dataset versions despite their differing schemas.

FilePurpose
loaders.pyReads the raw CSV log files (logon, device, email, http, file, LDAP, psychometric) into pandas DataFrames.
adapters.pyThe version-adapter layer. Each CERT version (r1–r6.2) has slightly different columns/encodings; adapters detect the version by "fingerprint" and translate it into the one canonical schema (strip DTAA/ prefix, normalize Insert→Connect, add missing headers, etc.).
pipeline.pyOrchestrates the whole ingest: load → adapt → combine → hand off. The end-to-end "universal pipeline."
feature_store.pyCaches computed results to Parquet (via pyarrow) so the expensive ingest/feature step runs once, not every request.
answers.pyLoads the ground-truth labels (insiders.csv — who is an insider, when, which scenario) so models can be trained/evaluated.

M1 features/ — daily behavioral fingerprints Python

backend/innersight/features/

Turns the cleaned logs into the 18 daily numbers per person — the input every model consumes.

FilePurpose
features.pyComputes the 18 daily features per user (logon counts, after-hours logons, USB copies, external emails, large attachments, job-site visits, cloud-upload visits, etc.) using pandas group-by + datetime logic.
temporal_features.pyBuilds the 28-day sliding windows (stride 7) out of those daily features — the time-series format the Temporal CNN reads.
The per-user statistical baseline (EMA + z-scores) — Module 1 in the slides — is the path that drives live demo scoring today. The deviation math itself lives close to features/scoring; the trained models below are the "powerful but not-yet-trained" path.

M2M3M4 models/ — the three neural networks Python

backend/innersight/models/

The deep-learning core. Know how these connect: CNN runs first → its embedding seeds the graph → GNN refines it → fusion concatenates everything → an MLP head outputs the risk score. A separate plain MLP is the baseline to beat.

FilePurpose
mlp.pyStandalone MLP baseline (~18→64→32→1). Sees one day at a time, no sense of time. Its job is to be the floor the fancier models must beat. Not fused into the final model.
temporal_encoder.pyTemporal CNN (TemporalPatternEncoder). Causal, dilated convolutions slide across the 28-day window to detect patterns of change (escalation, bursts). Outputs a 128-dim embedding + an attention map (which days mattered → interpretability).
graph_builder.pyBuilds the interaction graph from the logs: 4 node types (user, pc, url, file) and 5 edge types + reverses = 10. Seeds user-nodes with placeholder zeros that the CNN embeddings later replace.
graph_schema.pyDefines the graph's typed structure (node/edge type definitions and their feature layouts).
graph_encoder.pyGNN (GraphContextEncoder, GATv2). Attention-based message passing refines each user's temporal embedding with relational context (e.g. sharing a copied file with someone emailing a competitor). Outputs a 128-dim embedding.
fusion_model.pyThe full model (InsiderThreatDetector). Concatenates CNN(128) + GNN(128) + static(16) = 272-dim vector, then a classification head 272→128→64→1 outputs the final risk score. (Static = OCEAN personality 5 + role 8 + department 3.)
losses.pyFocal loss (α=0.75, γ=2.0) and friends — chosen because positives are ~0.4% of data, so plain accuracy is useless.
baseline.pyNon-neural baselines / the statistical baseline scaffolding used for comparison.
dataset.pyPyTorch Dataset/DataLoader wrappers that feed windows + labels to the training loop.

M1 scoring/ — risk scores & look-alikes Python

backend/innersight/scoring/

FilePurpose
scoring.pyTurns deviations (z-scores) or model outputs into a 0–1 risk score, applies the alert threshold, and assembles the explainable top-contributing-feature list.
suspect_discovery.pyStage 5 / look-alike search. Given one suspect, runs k-nearest-neighbor search over behavioral embeddings in Qdrant to find users behaving similarly.

M2M3M4 training/ — train & evaluate Python

backend/innersight/training/

FilePurpose
trainer.pyThe training loop: optimizer, epochs, focal loss, checkpointing — produces the .pt weight files.
evaluation.pyComputes metrics, above all AUPRC (the right metric for 0.4% positives), plus confusion matrix, precision/recall used on the Training and Model Comparison pages.
cross_version.pyTrain on one CERT version, test on another (generalization test, e.g. train r4.2 → test r6.2).

④ backend/tests/ the test suite

30+ test files, the source of the "202 tests passing" claim. The pattern is one test_X.py per source module, plus integration tests that exercise whole pipelines. Strong material to point at when defending engineering quality.

GroupExamples & what they verify
Unit teststest_adapters.py, test_features.py, test_mlp.py, test_temporal_encoder.py, test_graph_encoder.py, test_losses.py, test_scoring.py — each checks one module in isolation.
Integration teststest_integration_pipeline.py, test_integration_temporal.py, test_integration_graph.py, test_integration_fusion.py — check that stages work together end-to-end.
API / infratest_api.py, smoke_test_api.py (the API responds correctly), test_qdrant.py (vector search), test_reproducibility.py (seeds give identical results).
conftest.pyShared pytest fixtures (sample data, temp dirs) reused across all tests.

⑤ backend/checkpoints/ model weights & demo state

Where trained models and runtime state are saved. Generated — these are outputs, not source. Mounted into the backend container as /models.

ItemWhat it is
*.ptPyTorch weight files: best_model.pt, gnn_embeddings.pt, node2vec_embeddings.pt, metapath2vec_embeddings.pt, standardizer.pt, etc. If a relevant .pt is missing, api.py falls back to the statistical baseline and logs "No trained model found."

Exam-relevant: the presence/absence of files here is exactly what separates "runs live on the statistical baseline today" from "the trained deep models." If the teacher runs the demo with no fresh .pt, the Model Comparison / attention views show illustrative structure, not trained numbers.

One-paragraph recap to say out loud.

"The repo has three real parts. backend/innersight/ is the Python package, laid out as the pipeline runs: data/ ingests and normalizes the CERT logs, features/ turns them into 18 daily numbers, models/ holds the MLP baseline plus the Temporal CNN, the GATv2 GNN and the fusion model, scoring/ produces the risk score and the Qdrant look-alike search, and training/ trains and evaluates them; api.py exposes it all over Flask. frontend/src/ is the React UI — one folder per page — that just calls that API. data/synthetic_demo/ is the bundled dataset it runs on. Everything else — Makefile, docker-compose, tests, checkpoints, docs, CI — builds, runs, verifies or documents those three."
